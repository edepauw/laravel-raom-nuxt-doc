# Relations

Declare typed relations as property initializers. No `@Relation` decorator needed — relation builders self-register their metadata at bootstrap time.

## Types de relations

Laravel REST API Object Mapper supports four relation types:

- **`HasMany`** — one-to-many (e.g., User has many Posts)
- **`BelongsTo`** — inverse of HasMany (e.g., Post belongs to User)
- **`HasOne`** — one-to-one (e.g., User has one Profile)
- **`BelongsToMany`** — many-to-many via pivot table (e.g., User has many Roles)

**Morph relations** (polymorphic) are planned for a future release.

## HasMany

One parent, many children.

### Définir une relation HasMany

```ts
import {
  Model,
  Resource,
  Field,
  Key,
  HasMany,
} from "laravel-rest-api-object-mapper/runtime";

@Resource("users")
class User extends Model {
  @Key() @Field() id!: number;
  @Field() name!: string;

  posts = HasMany(() => Post, "posts");
}

@Resource("posts")
class Post extends Model {
  @Key() @Field() id!: number;
  @Field() title!: string;
  @Field() userId!: number;
}
```

**Syntax:** `HasMany(() => RelatedModel, 'relationName')`

### Accéder aux éléments

```ts
const user = await User.query().include("posts").findByKey(1);
const posts = user.posts.get(); // Post[]

console.log(posts.length);
console.log(posts[0].title);
```

**Note:** You must `include()` the relation in the query, otherwise `get()` returns an empty array.

## BelongsTo

Inverse of HasMany — child references parent.

### Définir une relation BelongsTo

```ts
@Resource("posts")
class Post extends Model {
  @Key() @Field() id!: number;
  @Field() title!: string;
  @Field() userId!: number;

  author = BelongsTo(() => User, "author");
}
```

**Syntax:** `BelongsTo(() => RelatedModel, 'relationName')`

### Accéder à l'élément parent

```ts
const post = await Post.query().include("author").findByKey(1);
const author = post.author.get(); // User | null

console.log(author?.name);
```

`get()` returns the parent instance or `null` if not loaded.

## HasOne

One-to-one relation.

### Définir une relation HasOne

```ts
@Resource("users")
class User extends Model {
  @Key() @Field() id!: number;
  @Field() name!: string;

  profile = HasOne(() => Profile, "profile");
}

@Resource("profiles")
class Profile extends Model {
  @Key() @Field() id!: number;
  @Field() bio!: string;
  @Field() userId!: number;
}
```

**Syntax:** `HasOne(() => RelatedModel, 'relationName')`

### Accéder à l'élément

```ts
const user = await User.query().include("profile").findByKey(1);
const profile = user.profile.get(); // Profile | null

console.log(profile?.bio);
```

## BelongsToMany

Many-to-many via pivot table.

### Définir une relation BelongsToMany

```ts
@Resource("users")
class User extends Model {
  @Key() @Field() id!: number;
  @Field() name!: string;

  roles = BelongsToMany(() => Role, "roles");
}

@Resource("roles")
class Role extends Model {
  @Key() @Field() id!: number;
  @Field() name!: string;

  users = BelongsToMany(() => User, "users");
}
```

**Syntax:** `BelongsToMany(() => RelatedModel, 'relationName')`

### Table pivot

The pivot table (e.g., `role_user`) is managed automatically by the Laravel backend. No explicit model needed.

### Accéder aux éléments

```ts
const user = await User.query().include("roles").findByKey(1);
const roles = user.roles.get(); // Role[]

roles.forEach((role) => console.log(role.name));
```

## Relations polymorphes (Morph)

**Planned for Phase 3.**

Morph relations (`morphTo`, `morphMany`, `morphOne`) will allow a model to belong to multiple parent types.

Example use case: Comments that can belong to Posts or Videos.

## Chargement de relations

### Eager loading

Load relations upfront with `include()`:

```ts
const users = await User.query().include("posts").include("profile").get();
```

All users are hydrated with their posts and profiles in one request.

### Lazy loading

**Not yet implemented.**

In the future, you'll be able to load relations on-demand:

```ts
const user = await User.query().findByKey(1); // no includes
await user.posts.load(); // load posts lazily
```

For now, always use `include()` in your queries.

### Relations imbriquées

Nest includes for deep loading:

```ts
const users = await User.query()
  .include("posts", (q) => q.include("comments", (c) => c.include("author")))
  .get();
```

Loads users → posts → comments → authors.

### Contraintes sur les relations

Apply filters and sorts to included relations:

```ts
const user = await User.query()
  .include("posts", (q) =>
    q.where("published", "=", true).orderBy("created_at", "desc").limit(5),
  )
  .findByKey(1);
```

Only published posts, latest 5, sorted by creation date.

## Exemple complet

```ts
import {
  Model,
  Resource,
  Field,
  Key,
  HasMany,
  BelongsTo,
  HasOne,
  BelongsToMany,
} from "laravel-rest-api-object-mapper/runtime";

@Resource("users")
class User extends Model {
  @Key() @Field() id!: number;
  @Field() name!: string;
  @Field() email!: string;

  posts = HasMany(() => Post, "posts");
  profile = HasOne(() => Profile, "profile");
  roles = BelongsToMany(() => Role, "roles");
}

@Resource("posts")
class Post extends Model {
  @Key() @Field() id!: number;
  @Field() title!: string;
  @Field() content!: string;
  @Field() userId!: number;

  author = BelongsTo(() => User, "author");
  comments = HasMany(() => Comment, "comments");
}

@Resource("comments")
class Comment extends Model {
  @Key() @Field() id!: number;
  @Field() content!: string;
  @Field() postId!: number;
  @Field() authorId!: number;

  post = BelongsTo(() => Post, "post");
  author = BelongsTo(() => User, "author");
}

@Resource("profiles")
class Profile extends Model {
  @Key() @Field() id!: number;
  @Field() bio!: string;
  @Field() userId!: number;

  user = BelongsTo(() => User, "user");
}

@Resource("roles")
class Role extends Model {
  @Key() @Field() id!: number;
  @Field() name!: string;

  users = BelongsToMany(() => User, "users");
}

// Query with nested includes
const user = await User.query()
  .include("posts", (q) => q.include("comments", (c) => c.include("author")))
  .include("profile")
  .include("roles")
  .findByKey(1);
```

## Bonnes pratiques

### Always include relations you need

```ts
// ✅ Include the relation
const user = await User.query().include("posts").findByKey(1);
const posts = user.posts.get();

// ❌ Forget to include — get() returns []
const user = await User.query().findByKey(1);
const posts = user.posts.get(); // []
```

### Use constraints to limit data

```ts
// ✅ Only load what you need
User.query().include("posts", (q) => q.where("published", "=", true).limit(10));

// ❌ Load all posts (can be huge)
User.query().include("posts");
```

### Don't mark relations with @Field

```ts
// ❌ Wrong — relation builders aren't fields
@Field()
posts = HasMany(() => Post, 'posts')

// ✅ Correct — no decorator
posts = HasMany(() => Post, 'posts')
```

### Use arrow functions for lazy model resolution

```ts
// ✅ Prevents circular dependency issues
posts = HasMany(() => Post, "posts");

// ❌ Direct reference can cause circular dependency
posts = HasMany(Post, "posts");
```
