# Identity Map

One primary key = one in-memory instance. The identity map ensures consistency across your application.

## Qu'est-ce que l'Identity Map ?

An identity map is an in-memory cache that stores model instances indexed by their primary key.

When you query for a resource:

1. The identity map checks if an instance with that key already exists
2. If yes, it returns the existing instance
3. If no, it creates a new instance and stores it

**Result:** One primary key always maps to the same JS object in memory.

## Pourquoi utiliser une Identity Map ?

Without an identity map, multiple API calls can produce multiple JS instances for the same resource:

### Sans Identity Map

```ts
const user1 = await User.query().findByKey(1);
const user2 = await User.query().findByKey(1);

console.log(user1 === user2); // false ❌
```

`user1` and `user2` represent the same database record, but they're different objects. Mutations to `user1` won't be visible in `user2`.

### Avec Identity Map

```ts
const user1 = await User.query().findByKey(1);
const user2 = await User.query().findByKey(1);

console.log(user1 === user2); // true ✅
```

Now `user1` and `user2` are the **same object**. Mutations are visible everywhere.

**Benefits:**

- ✅ **Consistency** — mutations are visible across the entire app
- ✅ **Performance** — reuses instances, reduces memory usage
- ✅ **Predictability** — one source of truth per resource

## Comment fonctionne l'Identity Map ?

### Hydration et réutilisation d'instances

When hydrating API responses:

```ts
// First query — creates new instance
const user = await User.query().findByKey(1);

// Second query — reuses existing instance
const sameUser = await User.query().findByKey(1);

console.log(user === sameUser); // true
```

The hydration process:

1. Extract primary key from API response
2. Check identity map for existing instance
3. If exists, update fields on existing instance
4. If not, create new instance and add to map

**Important:** Fields are updated in-place. The object reference stays the same.

### Cohérence des mutations

Field mutations are tracked and visible everywhere:

```ts
const user = await User.query().findByKey(1);
user.firstname = "Jane";

// Query again — same instance, sees mutation
const sameUser = await User.query().findByKey(1);
console.log(sameUser.firstname); // 'Jane' ✅
```

This works because `user` and `sameUser` are the **same object**.

### Identity map across relations

Relations also benefit from the identity map:

```ts
// Load user with posts
const user = await User.query().include("posts").findByKey(1);

// Query for a post directly
const post = await Post.query().findByKey(5);

// If post 5 belongs to user 1, they're the same instance
console.log(user.posts.get().includes(post)); // true
console.log(user.posts.get().find((p) => p.id === 5) === post); // true ✅
```

## Gestion du cache

### Invalidation du cache

To clear the identity map for a specific model:

```ts
User.clearIdentityMap();
```

This removes all cached instances. The next query will create fresh instances.

To clear all models:

```ts
Model.clearAllIdentityMaps();
```

### Quand invalider le cache ?

You typically don't need to manually invalidate. The identity map updates automatically during hydration.

Invalidate manually when:

- **Long-running app** — prevent unbounded memory growth
- **Stale data concerns** — force fresh instances from the API
- **Testing** — reset state between tests

### Stratégies de cache

**Default behavior:** Instances stay in the identity map until manually cleared or garbage collected.

**Automatic invalidation** (planned for Phase 6):

- Time-based expiration
- Size-based eviction (LRU)
- Query-triggered revalidation

## Cas d'usage avancés

### Reactive UI updates

In a Vue/Nuxt app, mutations to identity-mapped instances are reactive:

```vue
<script setup lang="ts">
const user = await User.query().findByKey(1);

// Later, in another component or composable
const sameUser = await User.query().findByKey(1);
sameUser.firstname = "Jane"; // UI updates automatically ✅
</script>

<template>
  <div>{{ user.firstname }}</div>
  <!-- Shows 'Jane' -->
</template>
```

Because `user` and `sameUser` are the same object, Vue's reactivity system picks up the change.

### Optimistic UI with rollback

```ts
const user = await User.query().findByKey(1);

// Optimistic update
const originalName = user.firstname;
user.firstname = "Jane"; // UI updates immediately

try {
  await user.save();
} catch (error) {
  // Rollback on failure
  user.firstname = originalName;
}
```

### Sharing instances across composables

```ts
// composables/useCurrentUser.ts
export function useCurrentUser() {
  return User.query().findByKey(1);
}

// Component A
const user = await useCurrentUser();
user.firstname = "Jane";

// Component B — sees the same instance
const sameUser = await useCurrentUser();
console.log(sameUser.firstname); // 'Jane'
```

## Bonnes pratiques

### Don't bypass the query builder

```ts
// ✅ Use query builder — identity map aware
const user = await User.query().findByKey(1);

// ❌ Direct API call — bypasses identity map
const response = await $fetch("/api/users/1");
```

### Trust the identity map

```ts
// ✅ Trust that instances are reused
const user1 = await User.query().findByKey(1);
const user2 = await User.query().findByKey(1);
// No need to check — they're the same

// ❌ Unnecessary identity checks
if (user1.id === user2.id) {
  /* ... */
}
```

### Clear identity map in long-running apps

```ts
// Example: clear every 10 minutes to prevent memory bloat
setInterval(
  () => {
    User.clearIdentityMap();
  },
  10 * 60 * 1000,
);
```

### Use identity map for local state management

```ts
// Treat identity-mapped instances as local state
const user = await User.query().findByKey(1);

// Mutate locally, commit later
user.firstname = "Jane";
user.lastname = "Doe";

// Later, when ready
await user.save();
```

This pattern works well for forms and multi-step workflows.
