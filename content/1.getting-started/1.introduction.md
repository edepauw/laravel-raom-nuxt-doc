# Introduction

Laravel REST API Object Mapper is a TypeScript/Nuxt module providing an ORM-like developer experience for APIs exposed by [`lomkit/laravel-rest-api`](https://github.com/lomkit/laravel-rest-api).

## Qu'est-ce que Laravel REST API Object Mapper ?

This module applies ORM patterns to REST resources. Instead of SQL queries, your models communicate with an HTTP API, but the developer experience remains familiar:

- **Decorator-based domain models** — define resources with `@Resource`, `@Field`, and `@Key`
- **Typed relation builders** — `HasMany`, `BelongsTo`, `HasOne`, `BelongsToMany`
- **Expressive query builder** — fluent API for filters, sorts, includes, and pagination
- **Identity map** — one primary key = one in-memory instance, no duplicate objects
- **Dirty tracking** — field changes tracked automatically, committed on `save()`
- **Deferred relation mutations** — `attach`, `detach`, `sync`, `toggle`, `create` operations deferred until `save()`

## Pourquoi utiliser ce module ?

Traditional REST API clients often lead to:

- **Boilerplate repetition** — manually parsing responses, managing state
- **Inconsistent data** — multiple instances for the same resource
- **Complex mutations** — hard-to-track changes, error-prone rollback logic

Laravel REST API Object Mapper solves these problems:

✅ **Familiar ORM patterns** — business logic stays in models, infrastructure concerns are abstracted  
✅ **Type safety** — full TypeScript support with autocomplete for fields and relations  
✅ **In-memory consistency** — identity map ensures one instance per primary key  
✅ **Deferred mutations** — atomic commits, automatic rollback on failure  
✅ **Developer experience** — expressive API, less boilerplate, faster development

## Concepts clés

### Decorator-based models

Define models with TypeScript decorators:

```ts
@Resource("users", { limits: [1, 10, 25, 50] })
class User extends Model {
  @Key() @Field() id!: number;
  @Field() firstname!: string;
  @Field() lastname!: string;

  posts = HasMany(() => Post, "posts");
}
```

### Identity Map

Without an identity map, multiple API calls produce multiple JS instances for the same resource.

With the identity map:

- ✅ One primary key maps to one instance in memory
- ✅ Mutations are visible everywhere in the app
- ✅ Hydration reuses existing instances

### Query Builder

Chain filters, sorts, includes, and pagination:

```ts
const user = await User.query()
  .where("firstname", "like", "Eli%")
  .include("posts", (q) => q.orderBy("id", "desc"))
  .findByKey(1);
```

### Relation system

Relations are declared as typed property initializers:

```ts
posts = HasMany(() => Post, "posts");
author = BelongsTo(() => User, "author");
```

No `@Relation` decorator needed — relation builders self-register their metadata at bootstrap time.
